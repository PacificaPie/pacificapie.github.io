The Translation Layer Is the Product 翻译层才是产品

May 2026  ·  ~ 7 min read  ·  a postmortem on VibeSearch 2026 年 5 月  ·  阅读约 7 分钟  ·  VibeSearch 项目复盘

A few weeks ago I shipped a small thing: a chatbot that takes a free-form sentence like "something for crying in a Honda Civic at 11pm" and recommends songs. I called it VibeSearch. The pipeline is unglamorous — Spotify open data, fuzzy matching, ChromaDB, a LangChain RAG loop, a Streamlit demo. Nothing in it is novel.

But there's one piece of it I keep thinking about, because I think it's the actual product. And it's the piece that almost didn't happen, because it didn't look like AI work.

The product gap that started everything

The user-side observation was simple. People do not describe music using energy = 0.9, valence = 0.2, danceability = 0.4. They describe music like: “road trip in October,” “the kind of sad I want to wallow in,” “something Frank Ocean–coded.” Spotify's API exposes the first kind of vocabulary; the user only speaks the second.

Most of the existing “mood-based” recommenders cheat their way around this by hand-curating playlists for a small number of moods (“Happy,” “Sad,” “Workout”). That works until you say “crying in a Civic at 11pm,” and the system has nothing to do but pick the closest of the seven labels it knows about.

The interesting question, the one I built the project around, was: how do we get a vector retriever to actually meet the user halfway?

Pretending it's an embedding problem

The first instinct — and I went here too — is to say: it's a vector-similarity problem. Embed the user's query. Embed each song's metadata. Cosine the two. Done.

This almost works. It works well enough that the demo isn't embarrassing. But it has a quiet failure mode: when you embed {title: "...", artist: "...", energy: 0.9, valence: 0.2} as text, the model is given a row of a CSV. There is no story in it. The numbers are floating tokens with no semantic gravity. The retriever is doing its best, but you're asking it to read a spreadsheet and feel something.

The translation layer

The thing that actually changed the demo from “cute” to “oh, it gets it” was not the retriever, the LLM, or the prompt. It was a tiny rule engine — maybe 80 lines of code — that translated the numerical features into natural-language descriptors before they ever hit the embedder.

So energy = 0.9 doesn't get embedded as “energy 0.9.” It gets embedded as “high-energy, intense, driving, kinetic.” valence = 0.2 doesn't get embedded as “valence 0.2.” It becomes “melancholy, downcast, somber.” Multiply across the seven Spotify features and each song now has a paragraph the embedder can actually read.

That paragraph is the same vocabulary the user is using. The retrieval gap closes because we did the translation work for it.

The product wasn't the AI. The product was the bilingual dictionary.

Why I think this is a PM artifact, not an engineering one

Here's the part I want to make a bigger claim about.

The translation layer required exactly zero machine learning. It was a lookup table. You could build it by hand in an afternoon — if you knew which descriptors mattered. And that “if” is the entire job.

To write the rule from energy = 0.9 to “intense, driving, kinetic”, you have to know (a) how users actually talk about high-energy music, (b) which of the synonyms a retriever will treat as semantic neighbors, and (c) which of those synonyms have enough specificity to discriminate. That's a vocabulary-design problem. It is not an engineering problem. It is a product spec.

A PM who writes that spec well makes the engineering trivial. A PM who waves their hand at it and says “the model will figure it out” gets a system that returns the same five sad songs to every melancholy query, and nobody on the team can explain why.

Three things I'd argue from this

1. Most “AI” problems are translation problems in disguise.

Search systems, ranking systems, recommendation systems — they all live or die on whether the system's internal vocabulary matches the user's. Embeddings, transformers, LLMs are how we cross the gap. What the gap is is a question for whoever understands both vocabularies. That tends to be the PM.

2. The smartest part of the spec is usually the smallest.

The translation rule engine is 80 lines. The Spotify ingestion pipeline is 600. The RAG retrieval loop is 150. By line count, the rule engine is invisible. By product impact, it's the entire reason the demo doesn't feel hollow. When you write specs, look for the small surfaces with disproportionate leverage.

3. Build it once before you spec it for the third time.

I would not have spotted the translation-layer move from a whiteboard. I spotted it because I had a retriever that was returning the same Frank Ocean tracks for every sad query, and I was sitting in front of it at 1am on a Sunday wondering why. Strategy is more honest after you've built the thing once.

What I'd build next

A few things on the running list, in case anyone wants to fight about them:

  • Crowdsource the descriptor dictionary instead of hand-writing it. Show users two songs and ask “which is more X?” with X being a candidate descriptor. Calibrate the rule engine off the votes.
  • Make the translation layer per-user. “Driving” means something different for somebody whose top genre is hyperpop vs. country.
  • Treat negation explicitly. “Sad but not depressing” is the kind of query current systems break on, and it's fixable at the rule layer, not the model layer.

That's where I am. The code is on GitHub if you want to pick it apart; tell me where I'm wrong at ruoyang.zhang@yale.edu.

几周前我做了个小东西：一个 chatbot，输入一句话比如 "晚上 11 点在 Honda Civic 里哭的时候听的歌"，它给我推荐音乐。 叫 VibeSearch。整套链路并不光鲜 —— Spotify 开放数据、模糊匹配、ChromaDB、 一段 LangChain RAG、一个 Streamlit demo。其中没有任何新东西。

但有一块东西我一直在想，因为我觉得它才是真正的产品。 而它差点没被做出来，因为它看上去不像 "AI 的活"。

一切的起点是产品 gap

用户侧的观察很简单。没有人会用 energy = 0.9, valence = 0.2, danceability = 0.4 来描述音乐。 人会说："想要点适合十月公路旅行的"、"我想沉溺在那种悲伤里"、"有点 Frank Ocean 那种感觉"。 Spotify 的 API 提供了第一种词汇，用户却只说第二种。

市面上的 "情绪推荐" 大多绕过了这件事，靠人工策展几个固定情绪歌单 ("Happy", "Sad", "Workout")。这个方法在你说 "11 点在 Civic 里哭的歌" 时立刻穿帮 —— 系统能做的只是从它知道的七个标签里挑一个最近的。

我做这个项目想搞清楚的真问题是： 怎么让向量检索真的能接得住用户的话？

把它当成 embedding 问题（差点）

第一反应 —— 我也走过这条路 —— 是说：这是个向量相似度问题嘛。 把用户 query embed，把每首歌的 metadata embed，算 cosine，搞定。

这差不多能跑。跑得 demo 不至于让人尴尬。 但它有一个隐性失败模式：当你把 {title: "...", artist: "...", energy: 0.9, valence: 0.2} 作为文本 embed 进去， 模型拿到的是 CSV 的一行。里面没有故事。数字是飘在空中的 token，没有语义重力。 retriever 已经在尽力了，但你在让它读一个表格然后产生情绪。

翻译层

真正把 demo 从 "还行" 推到 "诶它居然懂" 的不是 retriever，不是 LLM，不是 prompt。 是一个非常小的规则引擎 —— 大概 80 行代码 —— 它把数值特征翻译成自然语言描述， 在被 embed 之前就先翻译好。

于是 energy = 0.9 不会被 embed 成 "energy 0.9"， 它被 embed 成 "高能量、强烈、富有冲击力、动感"。 valence = 0.2 不会是 "valence 0.2"，它变成 "忧郁、低沉、沉重"。把七个 Spotify 特征都过一遍， 每首歌就有了一段 embedder 真正能读懂的描述。

这段描述用的是用户嘴里的词汇。检索的 gap 因此被合上 —— 因为我们替它把翻译做了。

产品不是 AI，产品是那本双语词典。

为什么我觉得这是 PM 的活，不是工程师的活

下面这个观点我想稍微 push 一下。

这个翻译层完全没用到任何机器学习。它就是一张查找表。 你下午就能手写一份 —— 前提是你知道哪些描述词重要。 这个 "前提" 才是整个工作。

要写出 energy = 0.9 → "强烈、动感、有冲击力" 这条规则， 你得知道 (a) 用户实际怎么描述高能量音乐，(b) 这些同义词里哪些会被 retriever 当成语义近邻， (c) 哪些有足够的区分度。 这是一个 vocabulary design 问题，不是工程问题。它是产品 spec。

一个能写好这份 spec 的 PM 会让工程变得平凡。 一个挥挥手说 "模型会自己学到的" 的 PM 会得到一个无论用户说什么悲伤都返回同样五首歌的系统， 而团队里没人能解释为什么。

三个我想从这里推出去的观点

1. 大多数 "AI 问题" 其实是翻译问题。

搜索系统、排序系统、推荐系统 —— 它们的成败全在于 系统内部的词汇能不能对上用户的词汇。 Embedding、Transformer、LLM 是我们跨越这个 gap 的工具。 gap 到底在哪里，是给同时懂两边词汇的人回答的问题。这个人通常是 PM。

2. 最聪明的那一部分 spec 通常是最小的。

翻译规则引擎是 80 行。Spotify 数据 ingestion 是 600 行。RAG retrieval loop 是 150 行。 按代码行数算，规则引擎是隐形的。 按产品影响算，它是整个 demo 不空洞的全部理由。 写 spec 的时候，找那种"面积小、杠杆大"的位置。

3. 在你第三次写 spec 之前，先做一遍。

我从白板上看不出 "翻译层" 这一招。我看出它是因为我面前坐着一个 retriever， 它在每一个悲伤 query 上都返回同样的几首 Frank Ocean， 而我在某个周日凌晨一点，对着它发呆。 亲手做完一遍再谈策略，会更诚实一些。

下一步想做的

列几个还没动的方向，欢迎来吵：

  • 把描述词典从手写变成众包。给用户两首歌，问 "哪首更 X？"， X 是候选描述词。基于投票来校准规则引擎。
  • 让翻译层个人化。"动感" 对一个 hyperpop top genre 的人和一个 country 听众来说意思不一样。
  • 把否定显式建模。"悲伤但不要丧" 是当前系统会崩的 query， 而修这种问题的位置在规则层，不在模型层。

先写到这里。代码在 GitHub 上，欢迎指出我哪里想错了 —— ruoyang.zhang@yale.edu。
