The problem

The feature recommends supporting materials for an event — “event → materials.” The LLM's output quality lands directly in front of frontline users, who see the candidates and check the ones they want. Bad output means a bad selection experience, immediately.

My call

The core of prompt engineering isn't wording — it's evaluation discipline: every version v_n gets compared against v_(n-1) on a fixed sample set. And the framing matters: LLM recommendations aren't the standard answer; they're a candidate list a human can veto.

What shipped

  • Slimmed the prompt from a 100-line exhaustive spec to 65 lines of principles + few-shot.
  • Fixed example contamination — after annotating the reasoning path in each example and adding negative examples, the badcase disappeared 100%.
  • Added fallback rules for sparse scenarios.

Validation

103 real production events, 7 prompt versions, each evaluated version-over-version on the fixed sample.

What iterating taught me — five rules

  1. Never omit the reasoning path from an example.
  2. Don't pile up rules — by prohibition rule #8, the rules start to backfire.
  3. Don't guess the customer segment.
  4. Hard quantity constraints need supporting rules to hold.
  5. Normalization is an engineering problem — don't make the prompt carry it.

问题

这个功能做的是 "事件 → 配套物料" 推荐。LLM 的输出质量直接落到一线用户面前 —— 他们看到候选项、勾选想要的那几个。输出不好，勾选体验立刻变差。

我的判断

prompt 工程的核心不是措辞，是评估纪律： 每个版本 v_n 都在固定样本上与 v_(n-1) 做对照评估。 定位同样关键：LLM 推荐不是标准答案，是人工可否决的候选清单。

交付

  • prompt 从 100 行穷举式瘦身到 65 行 "原则 + few-shot"。
  • 修复示例污染 —— 给示例补上推理路径标注、加入反面示例之后， badcase 100% 消失。
  • 为稀疏场景加兜底规则。

验证

103 条真实生产事件、7 个版本迭代，每一版都在固定样本上做逐版本对照。

认知迭代 —— 五条心法

  1. 示例的推理路径不能省。
  2. 规则不能堆 —— 第 8 条禁止规则开始反噬。
  3. 客群不能猜。
  4. 硬数量约束需补足规则。
  5. 归一化是工程问题，不该让 prompt 扛。
