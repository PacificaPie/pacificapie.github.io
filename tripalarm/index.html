让好价叫醒你。

价格会波动，人的注意力不会。TripAlarm 是一个盯价 Agent：意图说一次，它 7×24 代守；到价推送叫醒你，下单永远由人拍板。

「从消费旅行到企业差旅——同一套“代守”范式。」

Problem

没人真的每天去查价

想去三亚，等一个好价——但"等"这个动作没有人真正执行：不是忘了查，就是查到了也说不清"这个价算不算好"。

出差场景里同样的事换了主语：行程定了，票价每天在变。员工没空盯价，公司在为"下单的时机"默默买单。

把"等好价"从一个心愿，变成一个可以托付的任务。
How it works

三步：立约 → 代守 → 叫醒

01 · CREATE

一句话立约

「帮我盯三亚 8月 人均 6000 内」——目的地、时间窗、预算、偏好，落成一张结构化闹钟卡。缺什么，Agent 只追问什么。

02 · WATCH

Agent 代守

每日扫价：多闹钟共享 SKU 去重后批量取价；总价 ≤ 预算 是唯一触发硬约束，偏好评分只用来在达标者里择优。

03 · WAKE

好价叫醒

命中即推送到你的 IM（飞书 / Telegram / ntfy），带价格区间和锁价建议；回 /lock 一键锁定——human-in-the-loop。

Enterprise transplant

移植到企业差旅，只换约束不换范式

差旅的特殊在于：预算不是个人偏好，而是公司政策——硬约束天然存在、天然结构化。这恰好是"硬约束触发 + 偏好择优"这套判定架构的主场。约束更多，问题反而更规整：窗口写死、政策机读，剩下的核心是一个带 deadline 的下单时机问题。

这一形态已落成可运行实体（skill 形态，立约 → 代守 → 审批三段脚本，全离线演示）：好价腿之外多一条 deadline 兜底腿——度假可以不去，出差必须成行，宁可原价、不可误机。

💡 为什么"盯价"比"订票"更该先做 Agent 化：订票是低频单次动作，人 5 分钟能完成；盯价是高频持续动作，人根本不会执行。Agent 的边际价值在"人做不到的持续性"上，而不是"人本来就会做的事"上。命中后把决策权交还给人（锁价 / 审批），恰好落在企业最关心的可控与合规上。
🎯 两个受益人，一套账：对员工，把"记得去订票"变成"等它叫我"——出行的心智负担归零；对公司，每一单都留下"为什么是这个价、为什么是这个时机"的可解释记录——省钱第一次变得可审计。
Positioning

放进市面坐标系再看

"盯价"不是新念头——但已有的三种做法，各自留了一块空。

GOOGLE FLIGHTS · 信息透明 追踪 + 历史区间 + 价格保证。

免费、量大、够透明——但只提醒不执行，且没有"政策"概念；价格保证仅在算法有信心时给出、限定市场。

HOPPER · 预测金融化 Price Freeze：把盯价卖成期权。

小额押金锁价最长 21 天，涨了它补、跌了随行——预测能力变成了收入。但预测是黑盒，准确率为自报口径，无法审计。

NAVAN · 携程商旅 · 企业流程 Agent 改签退款自动化 + 合规内嵌。

企业差旅 agent 化最深的一批（专职 agent 军团 + 监督 agent）。但价值兑现从售后端开始——"事前省钱 / 下单时机"是整条链最薄的一环。

📍 我们的象限：约束驱动、可解释、审批内嵌的"订前时机层"。不预测未来——只依赖两个确定量（政策硬约束、出发 deadline），预测将来可以作为优化层叠上去，但不是价值成立的前提；每天一行扫价汇总就是审计线。诚实注：以上皆为规模化产品，本页是判定架构的可运行验证（V0，演示数据）。
Evaluation

怎么知道它盯得准？

代守型 Agent 的信任来自评测口径，不来自话术。三个问题，用固定历史价格样本做回归对照：

该叫没叫？

预算内的真实好价出现了，闹钟没响。

漏报率 · 直接伤信任
不该叫乱叫？

推送来了，点开发现并不划算或已过期。

误报率 · 三次狼来了就静音
叫得是时候吗？

命中价 vs 窗口内事后最低价的差距。

时机成本 · 择优质量
Design judgments

四个值得辩护的设计判断

TRIGGER ≠ SCORE 硬约束与评分分离。

触发命中只看「总价 ≤ 预算」一件事；偏好评分（如酒店权重）只用于在达标候选里择优。阈值混进评分，用户就再也说不清"它为什么响 / 为什么不响"。

OBSERVABILITY 没命中，也要发扫价汇总。

"没消息"和"没在扫"必须可区分——沉默的自动化系统会被用户默认为已经坏了。（这条纪律同样长在我的求职 agent 里：报警通道与数据通道分离。）

NO SILENT DROP 推送失败不置"已通知"。

通知发送失败时命中状态保持 active，下一轮扫价重试——好价宁可迟到，不可静默丢弃。异常不允许降级成合法的空。

COST ENGINEERING SKU 反向索引去重。

N 条闹钟盯同一航段 / 酒店时只查一次价。取价是最贵的外部调用，成本随用户数是否线性，架构在第一天就要给答案。

SCOPE DISCIPLINE V0 不做清单。

不做订票执行（那是 OTA / TMC 的执行层，我们是其上的时机层）；不做政策生成（政策是机读输入，不靠猜）；不做团行拼单。每条"不做"都有场景依据——收敛范围本身是产品判断，不是妥协。
